# Operator Console Product Architecture

## Product Goal

The console is a development upper-computer for a kitchen robot arm. Its job is to make simulation, teaching, waypoint management, controller tuning, and hardware debugging feel like one coherent workflow.

The console must always make three things visible:

- Which world is authoritative: simulation, real hardware, or both.
- Which control source is authoritative: GUI, joystick, planner, teaching drag, or safety controller.
- Which robot mode is active: homing, position control, gravity compensation, teaching, preview, execution, fault, or estop.

The GUI is not the safety authority. It is an operator interface and workflow coordinator. Real safety arbitration belongs in the robot control layer and hardware stop path.

## First Principles

### 1. State Has Provenance

Every displayed number should have a source:

- `sim`: generated by MoveIt, fake controller, MuJoCo, or another simulator.
- `real`: generated by motors, encoders, drivers, or hardware diagnostics.
- `estimated`: fused, filtered, or inferred from multiple signals.
- `operator`: entered or edited by the user.
- `stored`: loaded from local waypoint/teaching JSON.

The GUI should never show a joint angle without knowing whether it came from simulation or hardware.

### 2. Commands Need Authority

Many sources can ask for motion, but only one should own motion at a time:

- GUI sliders
- joystick/bluetooth controller
- MoveIt planner
- teaching drag mode
- homing state machine
- gravity compensation transition
- emergency stop or fault handling

All motion requests should go through an arbitration layer before they reach a controller or motor driver.

### 3. Preview Is Not Execution

Preview changes the simulated/planning state only.

Execution changes the active robot controller state. On real hardware, execution must require:

- connected hardware state
- homed robot
- correct control authority
- no estop
- no active joystick takeover unless explicitly allowed
- valid target inside limits

### 4. Mode Transitions Are Workflows

Mode changes are not just combo-box values. Some modes require ordered procedures.

Example: gravity compensation enable/disable.

1. Request transition.
2. Stop active teleop/planning command source.
3. Home or move to a safe calibration pose.
4. Disable motors if required by the drive layer.
5. Switch controller mode.
6. Re-enable motors.
7. Verify mode and telemetry.
8. Mark transition complete or failed.

The GUI should show this as an observable state machine, not a single button click.

## Core System State Model

Recommended top-level state:

```text
World:
  SimulationOnly
  HardwareConnected
  HardwareLive
  HybridMirror

RobotMode:
  Disconnected
  Idle
  Homing
  PositionControl
  VelocityControl
  Impedance
  GravityCompensation
  TeachingDrag
  PlanningPreview
  Executing
  Fault
  Estop

ControlAuthority:
  None
  GUI
  Joystick
  Planner
  TeachingDrag
  Homing
  Safety

DataSource:
  SimJointState
  RealJointState
  StoredWaypoint
  OperatorEdited
```

The GUI should display these states in the first viewport at all times.

## Simulation, Real, And Teaching Data Flow

### Simulation Preview

```text
GUI waypoint/sliders
  -> preview request
  -> simulated controller or MoveIt planning state
  -> /joint_states or planning scene
  -> RViz
```

This must not command physical motors.

### Real Execution

```text
GUI waypoint/sequence
  -> execute request
  -> arbitration/mode manager
  -> planner/controller action
  -> motor driver
  -> real joint states
  -> GUI and RViz
```

### Teaching Drag

```text
Operator physically drags arm
  -> gravity compensation / teaching controller
  -> real joint states
  -> GUI live pose
  -> optional mirror to RViz
  -> record trajectory/waypoints
```

Teaching data should store:

- joint positions
- timestamp
- source: `real_teaching`
- tool
- mode
- optional cartesian pose
- operator notes/tags

### Hybrid Mirror

Hybrid mirror means real hardware is the authoritative source, and simulation follows hardware:

```text
real /joint_states
  -> robot_state_publisher
  -> RViz
  -> GUI telemetry
```

The GUI should show `World: HardwareLive` or `World: HybridMirror` so the operator knows RViz is following the real arm.

## UI Information Architecture

The current tabs are useful during prototyping, but too fragmented for a real operator console.

Recommended next layout:

### Always Visible Header

Show compact status chips:

- World: Sim / Real / Hybrid
- Hardware: disconnected / connected / live / fault
- Homing: pending / homing / homed / failed
- Mode: position / gravity comp / teaching / executing
- Authority: GUI / joystick / planner / safety
- E-stop

### Main Workbench

One primary workbench should combine:

- joint sliders
- current joint table
- selected waypoint
- preview button
- plan/execute button
- RViz/MuJoCo status
- source indicators

This avoids forcing the operator to jump between Teleop, Waypoints, Monitor, and Visualization for one workflow.

### Side Panels

Use collapsible or tabbed side panels for:

- waypoint library and sequence editor
- teaching recording
- motor diagnostics
- PID tuning
- tool identification
- logs

### Mode Workflows

Special modes should have guided workflow panels:

- Homing
- Gravity compensation enable/disable
- Teaching drag capture
- To-real execution

Each should show prerequisites, active step, success/failure, and recovery actions.

## ROS2 Interfaces To Add

### System State

Topic:

```text
/arm/system_state
```

Should include:

- world
- robot_mode
- control_authority
- hardware_connected
- hardware_enabled
- homed
- estop
- faults
- active_command_source

### Authority Request

Service:

```text
/control_authority/request
```

Example requests:

- GUI wants preview authority
- GUI wants execute authority
- joystick wants teleop authority
- teaching wants drag authority

### Mode Request

Service/action:

```text
/mode_request
```

Modes:

- position
- impedance
- gravity_compensation
- teaching_drag
- homing

Gravity compensation should likely be an action because it has multiple steps.

### Waypoint Execution

Action:

```text
/execute_waypoint
/execute_sequence
```

The GUI should eventually call these instead of directly sending `/arm_controller/follow_joint_trajectory`.

## Next Implementation Milestones

### Milestone 1: State Clarity In GUI

- Add always-visible World / Hardware / Mode / Authority indicators.
- Distinguish sim joint state and real joint state in the model.
- Make Preview and Execute visually distinct.

Initial implementation status:

- The header now shows `World`, `Hardware`, `Homing`, `Mode`, `Authority`, backend, tool, contact, and E-stop.
- `RobotState` has explicit fields for `world`, `hardware`, `homed`, and `control_source`.
- Mock mode reports `World: SimulationOnly` and `Hardware: Mock`.
- ROS2 mode reports `World: SimulationOnly` and `Hardware: ROS graph` until a future `/arm/system_state` source is connected.

### Milestone 2: Unified Workbench

- Merge Teleop, Monitor, Waypoints, and Visualization status into one main workbench.
- Keep PID, diagnostics, tools, and logs as secondary panels.

### Milestone 3: Teaching Mode

- Add Teaching Drag workflow.
- Record real hardware joint states into waypoints/trajectories.
- Mirror real pose to RViz.
- Save source metadata with each waypoint.

### Milestone 4: Gravity Compensation Workflow

- Add gravity compensation transition panel.
- Represent transition as steps: request, home/safe pose, disable, switch, enable, verify.
- Prevent joystick/GUI/planner commands during transition.

### Milestone 5: To-Real Execution

- Add explicit Sim Preview / To Real controls.
- Require hardware connected, homed, no faults, and GUI authority.
- Route execution through a real control/planning action.
