# AGENTS.md

## Project Purpose

This repository is a Qt/PySide operator console prototype for a 5-DOF kitchen robot arm. It should support robot state monitoring, homing status, manual/joystick teleoperation, waypoint and teaching trajectory capture, motor diagnostics, PID tuning, tool identification, and future ROS2/RViz/MuJoCo integration.

The GUI is an operator console. It must not become the final safety authority. Safety-critical arbitration belongs in the robot control layer and hardware-level stop path.

## Repository Layout

- `kitchen_arm_gui.py`: repository-level launcher for the Qt/PySide console.
- `qt_operator_console/`: Qt/PySide implementation.
- `data/`: local waypoint, tool, PID, and teaching data generated by the apps.

## Run Commands

Windows Qt/PySide console:

```powershell
python -m pip install -r requirements.txt
python kitchen_arm_gui.py
```

Short-path virtual environment for Microsoft Store Python or long-path PySide6 install failures:

```powershell
python -m venv C:\tmp\armqt
C:\tmp\armqt\Scripts\python.exe -m pip install --upgrade pip
C:\tmp\armqt\Scripts\python.exe -m pip install -r requirements.txt
C:\tmp\armqt\Scripts\python.exe kitchen_arm_gui.py
```

Static validation without GUI launch:

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
python -c "import ast, pathlib; [ast.parse(p.read_text(encoding='utf-8')) for p in pathlib.Path('.').rglob('*.py') if '__pycache__' not in p.parts]; print('ast ok')"
```

On Ubuntu with ROS2:

```bash
source /opt/ros/$ROS_DISTRO/setup.bash
source install/setup.bash
python kitchen_arm_gui.py --backend ros2
```

## Architecture Rules

- Keep UI, runtime state, persistence, backend communication, and robot behavior separate.
- UI code should call backend methods such as `request_mode`, `start_homing`, `send_joint_target`, `write_pid`, and `identify_tool`.
- Backend adapters own ROS2 publishers, subscribers, service clients, and action clients.
- Keep a mock backend available on Windows so layout and workflows can be developed without ROS2.
- Do not publish raw motor commands from ordinary UI paths unless the backend explicitly represents a manual-control command interface.
- Emergency stop UI may request or latch stop state, but the actual safety authority must exist below the GUI.
- Homing must be represented as observable state machine status, not only as a button click.
- Planning and waypoint execution should be action/service based. The GUI may store local waypoints, but execution belongs to the planning/control layer.
- Prefer structured JSON or ROS message types over ad hoc strings for command/state payloads.

## ROS2 Interface Intent

- Monitor: subscribe `/joint_states`, motor diagnostics, `/arm/system_state`.
- Homing: subscribe `/homing_status`, call `/homing/start`.
- Mode: call `/mode_request`.
- Teleop: publish `/teleop/cartesian_cmd` or `/teleop/joint_cmd`.
- Waypoints: store locally first; execute with `/execute_waypoint` or `/execute_sequence` action.
- PID tuning: call `/motor_controller/set_pid`.
- Step test: use `/motor_controller/step_test` action.
- Tool identification: use `/tool_identification` action.
- Visualization: keep RViz/MuJoCo external at first, or stream rendered frames into a GUI widget.

## RViz And MuJoCo Guidance

- For true RViz embedding, prefer Qt/PySide and embed `rviz_common::VisualizationFrame` or a Qt-compatible visualization component.
- For MuJoCo, prefer offscreen rendering to image frames or a controlled Qt OpenGL widget.
- Avoid brittle X11 window reparenting except for short-lived experiments.

## Coding Style

- Keep code ASCII unless an existing file deliberately uses non-ASCII.
- Use small, explicit classes for backend interfaces, state containers, and widgets.
- Avoid broad refactors outside the Qt console unless the task explicitly asks for them.
- Add comments only when they explain non-obvious architecture or safety reasoning.
- Do not remove generated user data in `data/` unless explicitly asked.

## Done Criteria

For GUI changes:

- The app imports cleanly.
- New widgets are connected to backend methods or intentionally marked as placeholders.
- Mock mode remains runnable without ROS2.
- README or local docs are updated when commands or architecture change.

For ROS2 work:

- Message/service/action names are documented.
- The ROS2 backend degrades cleanly when `rclpy` is unavailable.
- UI remains responsive while ROS2 callbacks run in background threads or executors.
